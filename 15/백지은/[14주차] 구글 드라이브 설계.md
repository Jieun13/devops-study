## 15. 구글 드라이브 설계

### 1단계. 문제 이해 및 설계 범위 확정

- 먼저 결정해야 하는 것
    - 가장 중요한 기능 : 파일 업로드/다운로드, 파일 동기화, 알림(notification)
    - 지원 클라이언트 : 모바일 앱 + 웹 앱 모두
    - 파일 암호화 필요 여부 : 필요
    - 파일 크기 제한 : 10GB
    - 일간 능동 사용자(DAU) : 천만(10million)


- 집중할 기능
    - 파일 추가 (drag-and-drop)
    - 파일 다운로드
    - 여러 단말에 파일 동기화 (한 단말에서 추가 시 다른 단말에도 자동 동기화)
    - 파일 갱신 이력 조회(revision history)
    - 파일 공유
    - 파일이 편집/삭제/새로 공유되었을 때 알림 표시


- 비기능적 요구사항
    - **안정성** : 데이터 손실 발생 금지
    - **빠른 동기화 속도** : 동기화 지연 시 사용자 이탈 가능
    - **네트워크 대역폭** : 불필요한 대역폭 소모 금지 (특히 모바일 데이터)
    - **규모 확장성** : 대규모 트래픽 처리 가능
    - **높은 가용성** : 일부 서버 장애나 네트워크 문제에도 서비스 지속


- 개략적 추정치
    - 가입 사용자 : 5천만, DAU : 1천만
    - 모든 사용자에게 10GB 무료 저장공간 할당
    - 매일 각 사용자가 평균 2개 파일 업로드, 파일 평균 크기 500KB
    - 필요한 저장공간 총량 : 5천만 × 10GB = **500 페타바이트**
    - 업로드 API QPS ≈ 240, 최대 QPS ≈ 480

### 2단계. 개략적 설계안 제시 및 동의 구하기

단일 서버에서 시작하여 점진적으로 규모 확장하는 방식으로 접근

초기 구성 :
- 파일 업로드/다운로드를 처리할 웹 서버
- 사용자 데이터, 로그인 정보, 파일 정보 등 메타데이터 저장 DB
- 파일 저장을 위한 1TB 저장소 (drive/ 디렉터리)

제공 API :

| API | 설명 |
|---|---|
| 파일 업로드 API | 단순 업로드(소용량), 이어 올리기(resumable upload, 대용량/불안정 네트워크) |
| 파일 다운로드 API | path 인자로 다운로드할 파일 경로 전달 |
| 파일 갱신 히스토리 API | path와 limit 인자로 갱신 이력 반환 |

모든 API는 사용자 인증 필요, HTTPS 프로토콜 사용 (SSL 적용)


**이어 올리기(Resumable Upload) 3단계 절차**
1. 이어 올리기 URL을 받기 위한 최초 요청 전송
2. 데이터 업로드 및 업로드 상태 모니터링
3. 업로드 장애 발생 시 장애 발생 시점부터 업로드 재시작

**한 대 서버의 제약 극복**
- 저장공간 부족 → 데이터를 샤딩하여 여러 서버에 분산 저장 (user_id 기준 샤딩)
- 데이터 유실 위험 → 아마존 S3에 파일 저장, 다중화(같은 지역 내 + 여러 지역에 걸쳐)
- 로드밸런서, 웹 서버, 메타데이터 DB를 분리하여 SPOF 회피

**동기화 충돌 처리 전략**
- 먼저 처리되는 변경은 성공으로 처리
- 나중에 처리되는 변경은 충돌 발생으로 표시
- 두 파일 버전(로컬 사본 vs 서버 최신 버전)이 공존하게 되며, 사용자가 합치거나 대체 결정


**개략적 설계안 구성 컴포넌트**

| 컴포넌트 | 역할 |
|---|---|
| 사용자 단말 | 웹 브라우저, 모바일 앱 클라이언트 |
| 블록 저장소 서버 | 파일을 블록 단위로 나눠 클라우드 저장소에 업로드. 각 블록에 고유 해시값 할당 |
| 클라우드 저장소 | 파일을 블록 단위로 나눠 보관 |
| 아카이빙 저장소 | 오랫동안 사용되지 않는 비활성 데이터 저장 |
| 로드밸런서 | 요청을 모든 API 서버에 고르게 분산 |
| API 서버 | 사용자 인증, 프로파일 관리, 파일 메타데이터 갱신 등 담당 |
| 메타데이터 DB | 사용자, 파일, 블록, 버전 등 메타데이터 관리 (실제 파일은 클라우드에 보관) |
| 메타데이터 캐시 | 자주 쓰이는 메타데이터 캐싱 |
| 알림 서비스 | 이벤트 발생 시 클라이언트에게 알림 (발생/구독 프로토콜 기반) |
| 오프라인 사용자 백업 큐 | 오프라인 클라이언트를 위한 이벤트 정보 임시 보관 |

### 3단계. 상세 설계

### (1). 블록 저장소 서버

네트워크 대역폭 최적화 방법 두 가지 :
- **델타 동기화(delta sync)** : 파일 수정 시 변경된 블록만 동기화
- **압축(compression)** : 블록 단위로 압축. 텍스트는 gzip/bzip2, 이미지/비디오는 다른 알고리즘 사용

새 파일 업로드 시 블록 저장소 서버 동작 :
1. 주어진 파일을 작은 블록들로 분할
2. 각 블록을 압축
3. 클라우드 저장소로 보내기 전에 암호화
4. 클라우드 저장소로 전송

### (2). 높은 일관성 요구사항

- 강한 일관성(strong consistency) 모델 기본 지원 필요
- 같은 파일이 단말이나 사용자에 따라 다르게 보이는 것은 허용 불가
- 메모리 캐시는 최종 일관성(eventual consistency) 지원이 기본 → 강한 일관성 달성을 위한 추가 보장 필요
    - 캐시에 보관된 사본과 DB에 있는 원본이 일치
    - DB에 보관된 원본에 변경이 생기면 캐시의 사본 무효화
- 관계형 데이터베이스(ACID 보장)를 채택하여 높은 일관성 요구사항 대응

### (3). 메타데이터 데이터베이스 스키마

주요 테이블 :
- **user** : 사용자 기본 정보 (이름, 이메일, 프로파일 사진 등)
- **device** : 단말 정보 (push_id 포함). 한 사용자가 여러 단말 보유 가능
- **namespace** : 사용자의 루트 디렉터리 정보
- **file** : 파일의 최신 정보
- **file_version** : 파일 갱신 이력 (읽기 전용 레코드, 갱신 이력 보호)
- **block** : 파일 블록 정보. 특정 버전 파일을 블록을 올바른 순서로 조합하면 복원 가능

### (4). 업로드 절차

두 요청이 병렬로 전송됨 :

**파일 메타데이터 추가**
1. 클라이언트 1이 새 파일 메타데이터 추가 요청 전송
2. 메타데이터 DB에 저장, 업로드 상태를 대기 중(pending)으로 변경
3. 알림 서비스에 새 파일 추가 통지
4. 알림 서비스가 관련 클라이언트(클라이언트 2)에게 파일 업로드 중 알림

**파일을 클라우드 저장소에 업로드**
1. 클라이언트 1이 파일을 블록 저장소 서버에 업로드
2. 블록 저장소 서버가 파일을 블록 단위로 쪼갠 다음 압축 + 암호화 후 클라우드 저장소에 전송
3. 업로드 완료 시 클라우드 스토리지가 완료 콜백(callback) 호출 → API 서버로 전달
4. 메타데이터 DB에 해당 파일 상태를 완료(uploaded)로 변경
5. 알림 서비스에 파일 업로드 완료 통지
6. 알림 서비스가 관련 클라이언트(클라이언트 2)에게 업로드 완료 알림

### (5). 다운로드 절차

파일 변경 감지 방법 :
- 클라이언트 A가 접속 중이고 다른 클라이언트가 파일을 변경하면 → 알림 서비스가 변경 사실을 알리고 새 버전 끌어가도록 안내
- 클라이언트 A가 오프라인인 경우 → 데이터를 캐시에 보관했다가 접속 중으로 전환될 때 새 버전을 가져감

다운로드 흐름 :
1. 알림 서비스가 클라이언트 2에게 변경 알림
2. 클라이언트 2가 새로운 메타데이터 요청
3. API 서버 → 메타데이터 DB에 새 메타데이터 요청 및 반환
4. 클라이언트 2가 새 메타데이터를 받는 즉시 블록 다운로드 요청
5. 블록 저장소 서버 → 클라우드 저장소에서 블록 다운로드 후 클라이언트에 반환
6. 클라이언트 2는 전송된 블록으로 파일 재구성

### (6). 알림 서비스

- 파일 일관성 유지를 위해, 클라이언트가 로컬 파일 수정을 감지하는 순간 다른 클라이언트에 그 사실을 알려 충돌 가능성을 낮춰야 함
- 선택지 : **롱 폴링(long polling)** vs 웹소켓(WebSocket)
- 본 설계에서는 **롱 폴링** 채택 이유 :
    - 알림 서비스와 양방향 통신이 필요하지 않음 (서버 → 클라이언트 단방향)
    - 알림이 자주 발생하지 않고 단시간에 많은 데이터를 보낼 일이 없음

롱 폴링 동작 :
- 각 클라이언트가 알림 서버와 롱 폴링용 연결을 유지
- 특정 파일 변경 감지 시 연결을 끊고 메타데이터 서버와 연결하여 최신 내역 다운로드
- 다운로드 완료 또는 연결 타임아웃 시 즉시 새 요청을 보내어 롱 폴링 연결 복원

### (7). 저장소 공간 절약

파일 버전을 여러 데이터센터에 보관하면 저장용량이 빠르게 소진될 수 있음. 절약 방법 :

- **중복 제거(de-dupe)** : 계정 차원에서 중복된 파일 블록 제거. 두 블록이 같은 블록인지는 해시값 비교로 판단
- **지능적 백업 전략** :
    - 한도 설정 : 보관 파일 버전 개수에 상한을 두고, 도달 시 가장 오래된 버전 삭제
    - 중요한 버전만 보관 : 불필요한 버전과 사본 생성 방지
- **자주 쓰이지 않는 데이터를 아카이빙 저장소(cold storage)로 이동** : 아마존 S3 Glacier 같은 아카이빙 저장소 이용료는 S3보다 훨씬 저렴

### (8). 장애 처리

| 장애 유형 | 처리 방법 |
|---|---|
| 로드밸런서 장애 | 부(secondary) 로드밸런서 활성화, 박동(heartbeat) 신호로 상태 모니터링 |
| 블록 저장소 서버 장애 | 다른 서버가 미완료/대기 상태 작업 이어받음 |
| 클라우드 저장소 장애 | S3 버킷 여러 지역에 다중화, 다른 지역에서 파일 조회 |
| API 서버 장애 | 무상태 서버이므로 로드밸런서가 트래픽을 다른 API 서버로 우회 |
| 메타데이터 캐시 서버 장애 | 다중화된 다른 노드에서 데이터 조회, 장애 서버는 새것으로 교체 |
| 메타데이터 DB 주 서버 장애 | 부 서버 중 하나를 주 서버로 교체 |
| 메타데이터 DB 부 서버 장애 | 다른 부 서버로 읽기 연산 처리, 장애 서버는 새것으로 교체 |
| 알림 서비스 장애 | 한 서버 당 연결 수 1백만 개 이상. 장애 시 백만 명 이상이 롱 폴링 연결 재설정 필요 (느림) |
| 오프라인 사용자 백업 큐 장애 | 다중화. 큐 장애 시 구독 중인 클라이언트들은 백업 큐로 구독 관계 재설정 |

### 4단계. 마무리

- 이번 설계안의 핵심 두 부분 :
    - 파일의 메타데이터를 관리하는 부분
    - 파일 동기화를 처리하는 부분
- 알림 서비스(롱 폴링)를 통해 클라이언트로 하여금 파일 상태를 최신으로 유지
- 더 생각해보면 좋을 것
    - 블록 저장소 서버를 거치지 않고 파일을 클라우드 저장소에 직접 업로드하는 방안 (업로드 속도 향상 가능하나 분할/압축/암호화 로직을 클라이언트에 두어야 하는 단점, 보안 취약 가능성)
    - 접속 상태 관리 로직을 별도 서비스로 분리하여 다른 서비스에서도 활용 가능하게 하는 방안
