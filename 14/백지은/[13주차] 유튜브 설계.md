## 14. 유튜브 설계

### 1단계. 문제 이해 및 설계 범위 확정

- 먼저 결정해야 하는 것
  - 가장 중요한 기능 : 비디오 업로드 + 시청
  - 지원 클라이언트 : 모바일 앱, 웹 브라우저, 스마트 TV
  - 일간 능동 사용자(DAU) : 5백만(5million)
  - 평균 사용 시간 : 30분
  - 다국어 지원 : 필요
  - 지원 비디오 해상도 : 현존하는 대부분의 종류와 해상도
  - 암호화 필요 여부 : 필요
  - 비디오 파일 크기 제한 : 최대 1GB
  - 클라우드 서비스 활용 가능 여부 : 가능


- 집중할 기능
  - 빠른 비디오 업로드
  - 원활한 비디오 재생
  - 재생 품질 선택 기능
  - 낮은 인프라 비용
  - 높은 가용성과 규모 확장성, 안정성
  - 지원 클라이언트 : 모바일 앱, 웹 브라우저, 스마트 TV


- 개략적 규모 추정
  - DAU : 5백만
  - 사용자 1인당 하루 평균 5개 비디오 시청
  - 10%의 사용자가 하루에 1개 비디오 업로드
  - 비디오 평균 크기 : 300MB
  - 일일 필요 저장 용량 : 5백만 × 10% × 300MB = **150TB**
  - CDN 일일 비용 : 5백만 × 5비디오 × 0.3GB × $0.02 = **$150,000**

### 2단계. 개략적 설계안 제시 및 동의 구하기

시스템은 세 가지 컴포넌트로 구성됨

- **단말(client)** : 컴퓨터, 모바일 폰, 스마트 TV
- **CDN** : 비디오를 저장하고 재생 버튼 클릭 시 스트리밍 담당
- **API 서버** : 비디오 스트리밍을 제외한 모든 요청 처리 (피드 추천, 비디오 업로드 URL 생성, 메타데이터 DB/캐시 갱신, 사용자 가입 등)

면접관이 요청한 두 영역 :

### (1). 비디오 업로드 절차

업로드 설계 컴포넌트 :
- 사용자 단말
- 로드밸런서 : API 서버 각각으로 고르게 요청을 분산
- API 서버 : 비디오 스트리밍 외 모든 요청 처리
- 메타데이터 DB : 샤딩 + 다중화 적용
- 메타데이터 캐시 : 비디오 메타데이터와 사용자 객체 캐싱
- 원본 저장소 : 원본 비디오를 보관하는 BLOB 저장소
- 트랜스코딩 서버 : 비디오 포맷(MPEG, HLS 등) 변환
- 트랜스코딩 비디오 저장소 : 트랜스코딩 완료 비디오를 저장하는 BLOB
- CDN : 비디오 캐싱
- 트랜스코딩 완료 큐 : 트랜스코딩 완료 이벤트를 보관할 메시지 큐
- 트랜스코딩 완료 핸들러 : 큐에서 이벤트를 꺼내어 메타데이터 캐시/DB 갱신


업로드 흐름 (두 프로세스 병렬 수행) :

**프로세스 a. 비디오 업로드**
1. 비디오를 원본 저장소에 업로드
2. 트랜스코딩 서버가 원본 저장소에서 비디오를 가져와 트랜스코딩 시작
3. 트랜스코딩 완료 시 다음 두 절차 병렬 수행
  - 3a. 완료 비디오 → 트랜스코딩 비디오 저장소 → CDN 업로드
  - 3b. 완료 이벤트 → 트랜스코딩 완료 큐 → 완료 핸들러가 메타데이터 DB/캐시 갱신
4. API 서버가 단말에 업로드 완료 및 스트리밍 준비 완료 알림

**프로세스 b. 메타데이터 갱신**
- 원본 저장소에 파일 업로드되는 동안, 단말이 병렬로 비디오 메타데이터 갱신 요청을 API 서버에 전송
- 메타데이터에는 파일 이름, 크기, 포맷 등 포함
- API 서버가 메타데이터 캐시와 데이터베이스를 업데이트


### (2). 비디오 스트리밍 절차

- 비디오는 CDN에서 바로 스트리밍
- 사용자 단말에 가장 가까운 CDN 에지 서버가 비디오 전송 담당
- 전송지연 최소화

주요 스트리밍 프로토콜 :
- MPEG-DASH
- 애플 HLS
- 마이크로소프트 스무드 스트리밍
- 어도비 HTTP 동적 스트리밍(HDS)

### 3단계. 상세 설계

### (1). 비디오 트랜스코딩

트랜스코딩이 필요한 이유 :
- 가공되지 않은 원본 비디오(raw video)는 저장 공간을 많이 차지함
- 단말/브라우저마다 지원하는 비디오 포맷이 다름
- 네트워크 대역폭에 따라 고화질/저화질 비디오를 다르게 제공해야 함
- 모바일의 경우 네트워크 상황이 수시로 달라지므로 화질 자동/수동 변경 필요

인코딩 포맷 구성요소 :
- **컨테이너** : 비디오, 오디오, 메타데이터를 담는 파일 형식 (.avi, .mov, .mp4 등)
- **코덱** : 화질을 보존하며 파일 크기를 줄이는 압축/해제 알고리즘 (H.264, VP9, HEVC 등)


### (2). 유향 비순환 그래프(DAG) 모델

- 컴퓨팅 자원을 많이 소모하고 다양한 프로세싱 요구사항 지원 필요
- DAG 모델을 도입하여 작업을 단계별로 배열, 순차적/병렬 실행 가능하게 설계
- 원본 비디오는 비디오 / 오디오 / 메타데이터 세 부분으로 분리되어 처리

비디오 부분 적용 작업 :
- 검사(inspection) : 품질 확인, 손상 여부 확인
- 비디오 인코딩 : 다양한 해상도/코덱/비트레이트 조합으로 인코딩
- 섬네일(thumbnail) : 업로드 이미지나 자동 추출 이미지로 섬네일 생성
- 워터마크(watermark) : 비디오 식별정보를 이미지 오버레이 형태로 표시


### (3). 비디오 트랜스코딩 아키텍처

다섯 가지 주요 컴포넌트 :

| 컴포넌트 | 역할 |
|---|---|
| 전처리기 | 비디오 분할(GOP), DAG 생성, 데이터 캐시 |
| DAG 스케줄러 | DAG를 단계별로 분할하여 자원 관리자 작업 큐에 등록 |
| 자원 관리자 | 작업 큐 / 작업 서버 큐 / 실행 큐와 작업 스케줄러로 구성, 효율적 자원 배분 |
| 작업 실행 서버 | DAG에 정의된 작업 수행 (워터마크, 인코딩, 섬네일, 병합 등) |
| 임시 저장소 | 메타데이터는 메모리 캐시, 비디오/오디오 데이터는 BLOB 저장소 활용 |


### (4). 시스템 최적화

**속도 최적화 1 - 비디오 병렬 업로드**
- 비디오를 GOP 단위로 분할 후 병렬 업로드
- 일부 실패해도 빠르게 재개 가능

**속도 최적화 2 - 업로드 센터를 사용자 근거리에 지정**
- CDN을 업로드 센터로 이용하여 지역별 분산

**속도 최적화 3 - 모든 절차를 병렬화**
- 메시지 큐를 도입하여 시스템 결합도를 낮추고 병렬성을 높임
- 인코딩 모듈이 다운로드 모듈 완료를 기다릴 필요 없음

**안전성 최적화 1 - 미리 사인된 업로드 URL**
1. 클라이언트가 HTTP 서버에 POST 요청으로 미리 사인된 URL 수신
2. API 서버가 미리 사인된 URL 반환
3. 클라이언트가 해당 URL 위치에 비디오 업로드

**안전성 최적화 2 - 비디오 보호**
- DRM 시스템 도입 : FairPlay(애플), Widevine(구글), PlayReady(마이크로소프트)
- AES 암호화 : 암호화 후 재생 시에만 복호화
- 워터마크 : 소유자 정보를 이미지 오버레이로 삽입

**비용 최적화**
- 유튜브 비디오 스트리밍은 롱테일(long-tail) 분포를 따름
- 인기 비디오는 CDN을 통해 재생, 나머지는 비디오 서버를 통해 재생
- 인기 없는 비디오는 필요할 때만 인코딩
- 특정 지역에서만 인기 있는 비디오는 다른 지역에 옮길 필요 없음
- CDN을 직접 구축하고 ISP와 제휴하여 비용 절감


### (5). 오류 처리

- **회복 가능 오류** : 재시도로 해결 (트랜스코딩 실패 등)
- **회복 불가능 오류** : 작업 중단 후 클라이언트에 오류 코드 반환 (비디오 포맷 오류 등)

컴포넌트별 오류 처리 요약 :
- 업로드 오류 → 재시도
- 비디오 분할 오류 → 전체 비디오를 서버로 전송 후 서버가 분할 처리
- 트랜스코딩 오류 → 재시도
- 전처리 오류 → DAG 그래프 재생성
- DAG 스케줄러 오류 → 작업 재스케줄링
- 자원 관리자 큐 장애 → 사본(replica) 이용
- 작업 서버 장애 → 다른 서버에서 재시도
- API 서버 장애 → 다른 API 서버로 우회 (무상태 서버)
- 메타데이터 캐시 서버 장애 → 다른 노드에서 데이터 조회 후 교체
- 메타데이터 DB 서버 장애 → 주/부 서버 전환으로 대응

### 4단계. 마무리

- 더 생각해보면 좋을 내용
  - API 계층의 규모 확장성 : API 서버는 무상태 서버이므로 수평적 규모 확장 가능
  - 데이터베이스 계층의 규모 확장성 : 다중화와 샤딩 방법 논의
  - 라이브 스트리밍 : 비-라이브 시스템과 유사하나 응답지연 요구가 더 낮고 병렬화 필요성이 떨어짐. 오류 처리 방법도 달라야 함
  - 비디오 삭제(takedown) : 저작권 위반, 선정적, 불법 비디오는 업로드 과정에서 식별하거나 사용자 신고로 판별
